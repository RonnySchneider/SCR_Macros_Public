Installation and Updating

  • in order to use the macros for the first time you need to copy the macro folders manually into “C:\ProgramData\Trimble\MacroCommands\SCR Macros”
  • the name of that subfolder does now matter and must be “SCR Macros”
  • afterwards you need to restart TBC
  • up to TBC V5.81 > C:\ProgramData\Trimble\MacroCommands\SCR Macros
  • from TBC V5.9 onwards > C:\ProgramData\Trimble\MacroCommands3\SCR Macros
  • TBC V5.90 will change to a newer Python version with slightly changed syntax, hence all macros need to be altered, and TBC will look for them in “C:\ProgramData\Trimble\MacroCommands3\SCR Macros”

SCR_ExplodeAlignlabelToText

  •  TBC’s builtin “Explode” can’t explode Alignment Labels
  •  but it can be handy to use the Alignment Labeling to have some text nicely spaced along an Alignment

SCR_ExplodeIFC

  • explode an IFC-BIM object into Linework
  • can filter the triangles better than the Trimble version
  • the main direction is a zenith angle, in order to pick it on the screen click first a high point and then a lower point
  • the removal of duplicate lines happens per single mesh; if you explode multiple meshes, that touch each other, you'll most likely end up with duplicates at those borders
  • improved the speed of the duplicate line/vertice removal; vertices within 0.1 mm are considered the same
  • if a Limitbox is enabled it will create the linework that doesn’t cross it




SCR_ExplodeTextToLines

  • converts Texts to Linework
  • is used for better visualization in Propeller, Propeller shows CADText in a rather terrible way



  • also used in combination with SCR_SaveRestoreLabelPositions

SCR_ExifGPS

  • intention of the macro is it to make it easy to select just a certain amount of images of a drone flight, i.e. if only a certain amount can be uploaded/processed at a given time, or cost of processing tokens
  • it is advisable to run the macro in an empty project, no need to set a coordinate system, but doesn't hurt either
  • the macro will scan the selected folder recursively
  • it will extract the WGS coordinate that is saved in the image, convert it according to the TBC project settings into a local Easting/Northing coordinate and show the positions on the screen
  • if an image doesn't have a geotag at all, it will be placed on a separate layer at location 0,0,0; that way you can at least see the image name(s)
  • if the RTK-tag is lower than 50 (RTK fixed solution) the point is placed on a separate layer
  • it saves the image's path into the "Description 1" field, which is then used for the copy process
  • the RTK-tag is saved into the "Description 2" field
  • in the copy section of the macro you can quickly relayer/separate points/images
  • and once that is done you can copy the selected points/images to a temporary location before uploading them to Propeller




SCR_FixMediaLinks

  • if the field crew attaches images to their pickup points they can be shown in TBC, but after importing the JOBfile the image link TBC uses is relative to the TBC project folder
  • if you export the points via VCL into a different, i.e. master underground services project, the link won’t work anymore, the link is now relative to this project, that doesn’t have the images
  • the solution is to make the relative link an absolute link to the actual position of the image in the day jobs folder
  • the macro searches all subfolders of the selected root folder and tries to find a match with the original image link file name
  • the more you can narrow down that root folder, the quicker it will succeed
  • Update:
    you can now also define a target folder where all images will be copied to, i.e. if you want to hand them over with a VCL file

SCR_SaveRestoreLabelPositions

  • for easy/quick preperation of a new control file for Propeller
  • instead of moving labels and leaders around each time from scratch it will restore the ones it knows and highlights the ones that need attention



  • Here the TBC process:
    • open old TBC project from here Z:\2718 Survey Control\2718.0 ## Control registers ##
    • delete old labels and points
    • import the new control CSV
    • process feature codes
    • label points with "ANNO Point Labels"
    • run macro "SaveRestoreLabelPositions" restore positions of labels and leaders that have been processed before
    • check for new/magenta labels
    • after moving magenta labels, change color to ByLayer and rerun macro to save positions
    • select all labels and explode them twice with CAD > Explode
    • select all Leaders and use macro Explode CAD Leaders only do that AFTER the labels were exploded
    • select all texts and use macro Explode Text to Linework

SCR_SwitchSnapMode

  • best will be to apply a Keyboard-Shortcut like STRG+SHIFT+S to the macro
  • the macro DOESN'T affect Ortho-Snaps, only Object-Snaps
  • on start the macro will save current snap settings and "negate" what was enabled
  • this is based on the "No Snap" button in the macro

  • if that button was not toggled it will do that and set snaps to "Free" only, or otherwise sets this button to untoggled and restores previous snap settings
  • any change in the macro is immediate; no need for OK-button; hence you can immediately "Escape" out of it again
  • this way you can switch between Snaps/No-Snaps by just pressing some keys with one hand
  • as example:
    • press your shortcut -> snaps are disabled -> press Escape immediately and continue your work without snapping
    • next time you press your shortcut -> previous snaps are restored - > press Escape immediately and continue your work with snapping
  • the settings are saved with the project and not globally

SCR_DTMCleanup

  • !!! BETA version !!!
  • maximum tolerance values are limited to 1m for now
  • this macro runs through all the vertices of a surface, combines them if they fall within the set tolerance and builds a new surface
  • can fix issues with the Rainfall macro
  • on large surfaces, with lots of vertices it can take some time to finish – speed was drastically improved with 17/01/2022 update
  • currently boundaries for holes need to be added manually again
  • black lines are the original triangulation, red the new one



  • on very small scales the new triangulation can look a bit off, but that is a visualisation issue



SCR_DTMMerge

  • combine two surfaces
  • data from surface B will replace data from surface A
  • currently any applied boundaries are ignored
  • it creates a new surface with the data from surface A
  • offsets the edge breakline of Surface B the defined amount outwards
  • removes everything from the new surface that falls within that offset line
  • adds the content of surface B and retriangulates the resulting step
  • works better than TBC’s DTM combine and merge


  • if the two surfaces don’t overlap they will be combined, BUT the algorithm is determining this by using the edge breaklines both surfaces provide
  • they do not necessarily hug the visible triangulation, hence the surface might be falsely considered as inside the other when surface B takes precedence



  • in this example, surface A = the circle and surface B = large surface, the circle would disappear since it’s considered to be overwritten by surface B
  • if you swap it around, surface A = large surface and surface B = the circle, it will work since now the circle is considered higher priority
  • added option for manually entering a surface name
  • added option to overwrite an existing surface; i.e. if it's already tied to a volume computation you don't have to edit/fix that one

SCR_DTMSlopeColored

  • the macro folder contains a sample CSV and Surface Color Mapping file
  • highlight the areas of a DTM based on the triangle slope
  • 4 ways to do it
    • create a new DTM with flat triangles, but their elevation representing the slope of the originals

      • Pro:
        • working just with DTM data, in theory no large overhead
        • you can use the normal Surface Color Mapping
      • Con:
        • visual representation makes mostly sense in 2DPlanView
        • since we need near vertical steps the original triangles need to be shrunk and the triangulation process will create extra, near vertical ones; that leads to a
        • lot more triangles in the resulting surface and maybe performance issues
        • biggest con is that TBC’s Color Mapping algorithm isn’t 100% accurate, in the following example mapping is set to start with color blue at exactly 30.0, but
        • the surface with less than that is already blue



    • recolor the DTM triangle itself, is done by applying a material to each triangle
      • Pro: low overhead and quick when only a few different materials are used, but that leads to hard color steps
      • Con: if smooth colors are needed then a lot of materials with different colors need to be created, MSI manager and drawing might become slow

    • create closed and filled Linestrings
      • Pro: overhead stays the same no matter of using hard steps or interpolated colors
      • Con:
        • Linestrings have a large overhead; double up linework where triangles touch each other
        • 3D view can get pretty slow when used on large surfaces
    • create IFC Faces
      • Pro: lower overhead than Linestrings; and it also stays the same no matter of using hard steps or interpolated colors
      • Con:
        • no real way to export those manually created IFC objects
        • 3D view can get pretty slow when used on large surfaces

  • when using a CSV as input the format needs to be as follows, the header line must be included (but is information only)
    Slope[%],R,G,B
    0,0,255,0
    5,255,190,0
  • in case you use a CSV for recoloring a DTM with Materials a new category and materials will be created in the MSI Manager
  • when using the MSI Manager you need a Material Category, with materials where the name is the slope value (numbers only), and the color must be set accordingly

SCR_ElevateToPlane

  • is based on a discussion with a forum contact
    • assumption is a curved linestring
    • elevation information only defined at start and end of curve
    • the interpolated elevations along the curve will deviate from the cross fall plane
  • drape points or lines onto a 3-point plane without having to create a temporary surface
  • if the source line is a 2D CAD polyline it will create a new linestring
  • if the source line is already a linestring it will change it
  • it will keep the original horizontal definition, but all vertical information is removed and a new vertical profile is computed and applied
  • in order to strip all vertical information the macro will convert PointID definitions to coordinates
  • in case of segments that aren't straight line segments it will do the following
    • the start and end of the segment are draped onto the plane
    • the elevation in between is interpolated
    • the correct half-way position and its plane elevation are computed as well
    • if the difference between those two elevations is above the set threshold then additional information is added to the new vertical profile, otherwise the interpolation is considered good enough
    • in case of added vertical information this whole process is now reiterated with the neighboring information until the differences are below the threshold
    • this way it's only adding what is really necessary
    • the only time this approach probably fails is in the case of an ideal S-shaped curve, since the half-way point would sit exactly on the plane, and the iteration would stop

SCR_AnalyzeCloudAlongLine

  • manually select a bunch of lines (and normal CAD/Survey points if necessary)
  • the point cloud just needs to be visible on screen
  • you must either be in Plan or Cutting Plane View and PolygonSelect enabled
  • the macro will fake mouse gestures to select slivers of the point cloud
  • to be on the safe side this selection will be 50 % bigger
  • in a second step those selected points will be properly filtered to the defined Left/Right Offset
  • there are two modes
    • compute best fit 3D Polylines into the selected points

      • works best on clean clouds, without outliers
      • this is using a built in TBC function
      • the output will always be a chorded polyline
      • the chording tolerance defines how closely that polyline follows the computed polynomial function

    • compute average and/or median points at given intervals and areas around the line
      • the macro computes locations along each selected line
      • it then selects a rectangle of points around those locations (it does not check for overlaps with neighboring selections, usually that is preferred anyway)
      • in Cutting Plane View the means above/below the line and is translated into a pure elevation difference, there is no correction applied if the cutting plane view is tilted and not vertical
      • average is the plain average
      • for the median it compiles 3 lists with the Eastings/Northings/Elevations, sorts each of them independently and uses the value at the middle of each for
      • the resulting point
      • if the cloud contains a lot of outliers, i.e. small rocks/debris/dirt, it is possible to use a TargetDTM (no projected surface) to prefilter the points
      • Hint: run the macro multiple times
        • don’t use a DTM in the first step but use large combinevalues
        • create a first rough approximated surface from the resulting points
        • in a second run, with smaller combinevalues, you can now filter +/ along this rough DTM to ignore most of the outliers
        • of course, the cleaner the cloud is from the beginning the better the overall result will be

  • Example with a more or less horizontal deck scan cloud, last image shows the new surface created from median points



  • vertical wall example in Cutting Plane View



  • resulting points used to create projected surface and added contour lines

SCR_AddHogToLine

  • it computes a vertical hogprofile with the length of the given line and the specified hog (can be negative as well)
  • it then chords this hogprofile with the specified tolerance
  • it then computes positions/elevations on the original line at those chording locations
  • it creates a new line with original+profile elevation
  • there is also an option to draw/create the IsopachLine (Start/End Elevation = Zero) that has the arc instead of chorded segments
  • added option to use either arc or quadratic parabola for the hogprofile, with small hog values you’ll probably won’t see much difference
  • Note: when using the arc, if the hog value approaches the half way distance between start/end and hence is equal to the arc radius; the isopachlinestring visualization will be wrong; the reason is that in the internal computation of the profile geometry, like in 2D planview, an arc is defined by 3 points, start, end and hog point, using “chainage/elevation” as “X/Y”; for the linestring this is converted to chainage, radius, VPI; BUT if the diameter is getting equal to the profile length the arc tangents for the VPI get steeper and steeper and the VPI elevation approaches infinite
  • added options for linear or FergusonSpline distribution
  • added option for half arc/parabola
  • the FergusonSpline doesn’t support the horizontal/vertical/Nodespacing settings for the polyline approximation, instead it has its own value were “The segments that are going to represent the spline will not be further than this value.”
  • added option to keep original horizontal curves, and only change vertical profile component, or get them both chorded

  • creates “maximum rectangles” at chosen intervals around a bundle of lines
  • added option to create outline as 3DShell, start/end is closed and it will provide a valid volume



SCR_LinePerpVertOffset

  • with this macro you can offset the vertical profile of lines a chosen perpendicular amount
  • TIP: in combination with SCR_SubgradeFromFSL you can offset surfaces perpendicular; drop it only a very small amount, i.e. 0.000001 m, use the builtin “Project cleanup” to connect the line segments, use SCR_LinePerpVertOffset to offset them perpendicular and create a new surface




SCR_MeasureOffsetDistance

  • if you have Linework you can easily use “Explore Object” if you want to know the Distance/Offset along the line
  • if you only have points, you’d have to draw the line first, that’s were the macro steps in



  • computes now a simple vertical dH to the “P1P2Line”as well; i.e. if you have 3 points on a girder and want to know the hog
  • option added to draw the results as text


SCR_MergeLineProfile

  • lets you combine the horizontal component of one line (Line 1) with the vertical profile of another (Line 2)
  • the vertical profile can also be a 2DLine in Planview, in that case you’ll have to define the origin (Chainage 0.0 and Elevation 0.0; ChainageAxis is assumed parallel to Easting and ElevationAxis parallel to Northing)
  • in case the length of Line 1 doesn’t match the length of Line 2 you can compute the Scale to stretch/shrink it accordingly
  • the vertical scale will work independently of the horizontal scale
  • you can also draw the vertical profile to the plan view (with variable scales and direction)
  • in either case, if a scale other than 1.0 is applied the macro needs to chord the profile first, otherwise any curvature in the profile could be compromised, you can’t just simply stretch an arc or spline; it will use the bigger one of the scales to recalculate the chording tolerances



  • added smoothing function
    • for a basic 2D-Polyline you may have seen the “smoothing” in the properties pane
    • after some tinkering I found out what it is doing, it creates a parabola between the segment midpoints
    • BUT, the result can be rather coarse, and it’s not available for 3D stringlines
    • hence the macro, it’ll do the same for the horizontal and/or vertical profile of a stringline
    • it ignores any existing curves, it just looks at the nodes/kinks in the line; basically IP based
    • a stringline doesn’t know the element parabola, only alignments do, for now the resulting parabola is chorded before drawn
    • since the resulting horizontal alignment will be shorter than the original, the vertical profile needs to be scaled in chainage direction to match the new length, hence the macro needs to chord it as well in order to do that

  • added an option to change all segments of a line to either smooth or straight

SCR_MinimumRectangle

  • creates a rectangle with minimum area around a chosen outline




SCR_MultiOffset

  • I’m still working on the correct elevations for the offset lines, on lines with sharp kinks and elevation changes the computation is probably still not 100% correct – TBC’s standard offset doesn’t compute it right either
  • offset a bunch of lines a given offset
  • i.e. dozens of bearing pad outlines to create lines for the bearing recesses, or girder CL’s in order to be used in SCR_AnalyzeCloudAlongLine

SCR_PowerlineSway

  • comes from a TBC forum request
  • they were interested to see if swaying powerlines would cross property boundaries
  • it does simply rotate the polyline around the axis between start and end node, it does not simulate any gravity influence onto the cable
  • once the new line is created it is selected as the new source, that way you can use the rotation value as increment
  • it should work with all lines/alignments that are 3D, or a curved 2D line with its elevation set to 0.0
  • it won't work for lines with undefined elevation, and closed lines
  • the source line will automatically be approximated by a polyline. The lower you set the approximation values, the better the fit. That only matters if the source line has arcs/splines/parabolas. If the source line is a polyline the new line will match it perfectly.



SCR_SweepShape

  • sweep an outline along an alignment and use a second line to rotate it
  • in choosing an orientation/rotation point off the shape you can offset it if necessary
  • curves in the original outline will be approximated by a polyline, with “node tolerance” you can set how accurate it shall be
  • added option to create outline as 3DShell, currently it will be open at the start/end and won’t provide a volume

SCR_AutoConnectPoints

  • quickly connect a group of points with a linestring
  • newly added option to use PointID instead of XYZ
  • the macro will start at the first point and will jump to the next closest one



  • it includes a special function for near vertical bolts (the 2D distance to the second point of the bolt must be smaller than to any other selected point), it won’t work for horizontal bolts



SCR_CreatePointsOnXlines

  • computes XLines off a selected line and creates points where those cross a second line
  • added Inspection mode, makes it easier to check tables like this




  • added Bench-Pattern option, Lines will be drawn from high to low



SCR_P1_XY_P2_Z

  • concrete edges are usually chamfered
  • in the field it’s easier to pickup the edge itself and a point on top of the concrete
  • with this macro you can quickly create a point where the actual corner would be, it takes the E/N component of the 1st point and only the Z from the 2nd point to create a new one



  • you either pick two points manually or have the macro match the feature codes



  • use it in combination with SCR_AutoConnectPoints
  • added option to autoadvance the slider in Cutting Plane View to speed up extraction from Pointclouds



SCR_PasteCSVPoints

  • allows you to quickly create points via copy and paste of CSV-style data
  • i.e. from an Excel table or a plain text file out of i.e. Notepad++
  • the data doesn't necessarily have to include a point number; the bare minimum is X/Y
  • text from a PDF table should probaby by checked in Excel first

SCR_PointBlock2Point

  • convert CADPoints or Blocks into named TBC points
  • the macro first extracts the block insertion point or CADPoint coordinate/elevation (this first elevation is a fallback in case the next step fails)
  • if it afterwards can find another elevation in the layers/texts it will assume that one
  • if the Information like Point Name, Elevation, Attributes is on separate Layers inside the block use the option “Data is in block”



  • if the data is already exploded and the attributes are text entities at the same offsets from the points use the second option
  • you have to select a sample to show the macro at which offsets which attribute can be found around the point





SCR_PointFromVectorOffsetStation

  • define an axis with two points and create station/offset points along it
  • TIP: use +/ in the station offset fields to add up dimensions from a drawing
  • that way it is simple to compute i.e. the outline points of a headstock/abutment, they are usually based on one axis with square dimensions from it




SCR_PointSurfaceSlopeToCSV

  • the TBCPoint Label can only show the difference to a DTM, but it can’t show the surface slope at that point
  • (a SmartText can actually do it)
  • the macro retrieves the slope of the surface at the point locations and writes them into “\Downloads\PointSurfaceSlopes.csv” (that is hard-coded)
  • it can also add the slope as text on the screen
  • if you change the point or the surface the text will not update, it’s not a dependent label

SCR_SelectPtsOffLine

  • this macro selects all named coordinate points along all selected linestrings
  • it will highlight vertices that aren't point based
  • if a line is fully point based it'll be highlighted in basic green
  • intention of it is to make sure that a linestring is created by PointID instead of X/Y, that way the line will change if the point is changed
  • the result is also written into “\Downloads\NodeOutput.csv” (that is hard-coded)



SCR_SimpleCADPoint

  • often a simple CADPoint is enough as construction point, no need to create named points all the time
  • you can still use all the right-click commands like BearingDistance etc.



  • option for QuickMidPoint might be handy on circular IFC objects
